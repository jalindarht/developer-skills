# Code Review — HADV.OMS.Services

Reviews diffs, PRs, or files for logic bugs, security holes, performance issues, architecture violations, and deviations from the project's established patterns before code reaches production.

---

## Mandatory Rules

```
────────────────────────────────────────────────────────────────
  You are a senior .NET engineer working on HADV.OMS.Services.
  This is a .NET 8 Clean Architecture project using Ardalis
  libraries. Follow these rules on every review.

  1. UNDERSTAND BEFORE YOU REVIEW
     → Read the changed files AND similar existing files
     → Understand intent before judging implementation
     → Never assume — look at the actual codebase

  2. REUSE — NEVER DUPLICATE
     → Check if a similar service/spec/endpoint already exists
     → If shared logic is duplicated, flag it
     → Ask: "Does this already exist somewhere?"

  3. ENFORCE PROJECT PATTERNS
     → No controllers — Ardalis ApiEndpoints only
     → Services must return Result<T> or Result
     → Queries must use Specifications, not raw LINQ in services
     → Guard.Against.* for input validation in services
     → All async methods must accept CancellationToken

  4. NO AI TOOL REFERENCES — ANYWHERE
     → No "Generated by..." in code or comments
     → No "Co-Authored-By: Claude/Copilot" in commits
     → All output must look 100% human-written
────────────────────────────────────────────────────────────────
```

## Step 2: Review — 8 Passes

### Pass 1: Architecture & Pattern Compliance

```
────────────────────────────────────────────────────────────────
  PASS 1: ARCHITECTURE & PATTERN COMPLIANCE

  Does the code follow the project's established structure?

  ENDPOINTS (CentralAPI/Endpoints/[Feature]/)
  ☐ Inherits EndpointBaseAsync.WithRequest<TReq>.WithActionResult<TRes>
  ☐ One endpoint per file — no combined GET+POST in one class
  ☐ Request class in a separate [Endpoint].Request.cs file
  ☐ Route defined via constant in CentralAPI/Constants/ApiEndPoints.cs
  ☐ Has [SwaggerOperation] with Summary, OperationId, Tags
  ☐ Uses this.ToActionResult(result) — not Ok()/BadRequest() directly
    (exception: explicit error extraction like PlaceOrder)
  ☐ [Authorize] present on protected endpoints
  ☐ HTTP verb matches intent: GET-like reads use HttpPost per pattern,
    creates use HttpPost, updates use HttpPut/HttpPatch, deletes HttpDelete

  SERVICES (Core/Services/[Feature]/)
  ☐ Public interface defined with all methods returning Result<T>/Result
  ☐ Implementation class marked internal
  ☐ Registered in CoreExtensions.RegisterCoreServices() as Transient
  ☐ Guard.Against.* used for input validation before business logic
  ☐ Uses IRepository<T> — no direct DbContext injection in services
  ☐ Queries delegated to Specifications — no inline LINQ queries
  ☐ All async methods have CancellationToken parameter
  ☐ Returns Result.NotFound() / Result.Error() / Result.Success(value)

  SPECIFICATIONS (Core/Entities/Specifications/[AggregateRoot]/)
  ☐ Placed under the correct aggregate root folder
  ☐ Class marked internal
  ☐ Extends Specification<TEntity> or Specification<TEntity, TDto>
  ☐ Query built in constructor using Query.Where().Include().Select()
  ☐ No business logic inside a specification

  EF CONFIGURATIONS (Infrastructure/Data/Configurations/)
  ☐ Extends BaseEntityConfiguration<T> for BaseEntity types
  ☐ TableName and SqlSchema set using Constants.TableSchemas.*
  ☐ base.Configure(builder) called at the END
  ☐ New entity's DbSet added to AppDbContext
  ☐ Migration created for schema changes

  DTOs (Core/DTO/)
  ☐ Simple POCO — no base class, no inheritance
  ☐ Strings initialized to string.Empty
  ☐ Collections initialized with new() or left nullable
  ☐ Nullable types used where the value is optional

  ENTITIES (Core/Entities/)
  ☐ Extends BaseEntity (or LookupBaseEntity for lookup tables)
  ☐ Aggregate roots implement IAggregateRoot
  ☐ [MaxLength], [Required] applied where needed
  ☐ [Column(TypeName = "json")] on JSON column properties
  ☐ No business logic in entity constructors beyond initialization
────────────────────────────────────────────────────────────────
```

### Pass 2: Logic & Correctness

```
────────────────────────────────────────────────────────────────
  PASS 2: LOGIC & CORRECTNESS

  Does the code actually do what it claims?

  ☐ Does the service method logic match the endpoint's intent?
  ☐ Are null checks performed before accessing navigation properties?
    (EF navigation props can be null if not Included)
  ☐ Is Result<T> checked with .IsSuccess before using .Value?
  ☐ Are async/await used correctly? (missing await on async calls)
  ☐ Are LINQ operations correct? (.FirstOrDefault vs .First,
    .Any vs .Count > 0, .Select vs .Where)
  ☐ Is soft-delete respected? (IsDeleted global filter is automatic,
    but manually constructed queries may bypass it)
  ☐ Are DateTimeOffset.UtcNow used for timestamps — not DateTime.Now?
  ☐ Is the CancellationToken passed down to all async calls?
  ☐ Are return values from repository methods (AddAsync, UpdateAsync)
    used when the result is needed?

  COMMON BUGS
  → Navigation property accessed without Include in the specification
  → result.Value used without checking result.IsSuccess first
  → Missing await before async repository method
  → DateTime.Now instead of DateTimeOffset.UtcNow
  → Specification filtering on IsDeleted when global filter already handles it
  → Guard called after business logic instead of at the start
────────────────────────────────────────────────────────────────
```

### Pass 3: Security & Authorization

```
────────────────────────────────────────────────────────────────
  PASS 3: SECURITY & AUTHORIZATION

  Can this code be exploited or accessed without permission?

  AUTHENTICATION
  ☐ Endpoint has [Authorize] if it accesses user/tenant data
  ☐ Auth0 JWT validation relies on middleware — do not re-validate
    manually; just ensure [Authorize] is applied

  AUTHORIZATION
  ☐ If filtering by userId, is it taken from the JWT claim —
    not from the request body? (users cannot spoof their own ID)
  ☐ If filtering by locationId/tenantId, is there a check that
    the requesting user belongs to that location/tenant?
  ☐ Can a user mutate another user's data by changing an ID in the request?

  INPUT VALIDATION
  ☐ Guard.Against.* called in services for all required parameters
  ☐ [Required], [MaxLength] on Request class properties
  ☐ File uploads validated for type and size if applicable
  ☐ Enum inputs validated against allowed values

  DATA EXPOSURE
  ☐ Response DTO excludes sensitive fields (passwords, internal keys)
  ☐ Error messages don't leak stack traces or internal DB details
  ☐ Internal entity IDs not exposed when a public-facing key suffices

  RED FLAGS — STOP AND ESCALATE
  → Raw SQL string concatenation (use EF parameterized queries)
  → Hardcoded credentials or connection strings in source
  → [AllowAnonymous] on an endpoint that modifies data
  → User-supplied ID used directly in a query without ownership check
────────────────────────────────────────────────────────────────
```

### Pass 4: Error Handling

```
────────────────────────────────────────────────────────────────
  PASS 4: ERROR HANDLING

  What happens when things go wrong?

  SERVICES
  ☐ Guard.Against.* throws at the service boundary — OK, intentional
  ☐ Repository exceptions (DbUpdateConcurrencyException, etc.)
    caught where the service can handle them gracefully
  ☐ Not-found cases return Result.NotFound() — not null or exception
  ☐ Validation failures return Result.Error(errorCode, message)
  ☐ Error codes come from enums in Core/ErrorCodes/ — not raw strings

  ENDPOINTS
  ☐ this.ToActionResult(result) used so Result is converted consistently
  ☐ When bypassing ToActionResult (e.g. custom error extraction),
    the status code matches the failure type (400 vs 404 vs 500)
  ☐ No empty catch blocks — every catch either logs, returns an error,
    or re-throws

  CONCURRENCY
  ☐ DbUpdateConcurrencyException caught on cart/order mutation paths
  ☐ Idempotent operations treat concurrent completion as success

  ANTI-PATTERNS
  → catch (Exception) {} — silent swallow
  → Returning 200 OK from a catch block
  → Throwing raw exceptions from a service (use Result.Error instead)
  → result.Value accessed without checking IsSuccess (will throw)
────────────────────────────────────────────────────────────────
```

### Pass 5: Performance & Database

```
────────────────────────────────────────────────────────────────
  PASS 5: PERFORMANCE & DATABASE

  Will this be fast enough at scale?

  QUERIES
  ☐ Specification selects only needed fields via .Select() projection
    — not fetching full entities when only a DTO is needed
  ☐ No N+1: repository not called inside a loop
    (loop over items + repo call per item = N+1)
  ☐ .Include() used for navigation properties, not lazy loading
  ☐ Pagination applied to list queries (Skip/Take in specification)
  ☐ COUNT done with .CountAsync / .AnyAsync — not fetching all rows

  EF CONFIGURATION
  ☐ Indexes configured for columns used in frequent WHERE clauses
  ☐ JSON columns use [Column(TypeName = "json")] + owned entity or
    value converter — no raw string JSON parsing in services
  ☐ Cascade delete rules defined where needed

  BACKGROUND WORK
  ☐ Long-running operations offloaded to BackgroundService (CronJob/)
    — not running in the HTTP request pipeline
  ☐ IServiceScopeFactory used in background services — not Scoped
    services injected directly into a Singleton

  N+1 EXAMPLE (flag this pattern)
  BAD:
    foreach (var item in cartEntity.Items)
    {
        var product = await _productRepo.FirstOrDefaultAsync(...);
    }
  GOOD:
    Use a specification that Includes or projects the needed data
    in a single query.
────────────────────────────────────────────────────────────────
```

### Pass 6: Code Duplication

```
────────────────────────────────────────────────────────────────
  PASS 6: CODE DUPLICATION

  Is the same logic written more than once?

  ☐ Does a specification for this query already exist?
  ☐ Is similar Guard.Against.* validation duplicated across services?
  ☐ Is the same DTO-to-entity or entity-to-DTO mapping repeated?
    (belongs in the specification's Select() or a shared mapper method)
  ☐ Is the same error code string repeated instead of an enum value?
  ☐ Are similar endpoint structures copy-pasted instead of using
    a shared base request/response pattern?

  RULE OF THREE
  → Duplicated once? Acceptable short-term.
  → Three copies of the same logic? Extract it.
────────────────────────────────────────────────────────────────
```

### Pass 7: Naming & Readability

```
────────────────────────────────────────────────────────────────
  PASS 7: NAMING & READABILITY

  Can another developer understand this in 30 seconds?

  NAMING CONVENTIONS (C# 12 / project style)
  ☐ Classes: PascalCase — GetProductById, CartService
  ☐ Private fields: _camelCase with underscore — _cartRepo
  ☐ Parameters/locals: camelCase — userId, cancellationToken
  ☐ Interfaces: IPrefixed — ICartService
  ☐ Constants/enums: PascalCase — TableSchemas.Sale
  ☐ Booleans named as questions: isActive, hasPermission, canEdit
  ☐ No magic numbers — use named constants or enum values

  STRUCTURE
  ☐ Methods are focused — one responsibility per method
  ☐ Nesting depth under 3 levels (guard clauses flatten nesting)
  ☐ var used only when the type is obvious from the right side;
    explicit types preferred per project standard
  ☐ File-scoped namespaces used (namespace X.Y.Z; not nested braces)
  ☐ CRLF line endings (Windows, per project standard)
  ☐ 4-space indentation

  COMMENTS
  ☐ Comments explain WHY, not WHAT
  ☐ No commented-out dead code left in the file
  ☐ No TODO comments without a linked task
────────────────────────────────────────────────────────────────
```

## Step 3: Write the Review Report

### Finding Format

```
[SEVERITY] [Pass] — Short description

File: HADV.OMS.Core/Services/CartService.cs (line 45)
Issue: What is wrong and why it matters
Fix:   What to do instead (with code snippet when helpful)
```

**Severity levels:**
- 🔴 **CRITICAL** — Must fix before merge (logic bug, security hole, crash risk, data loss)
- 🟡 **WARNING** — Should fix before merge (performance issue, missing validation, pattern violation)
- 🔵 **SUGGESTION** — Nice to fix, not blocking (naming, readability, minor optimization)

### Report Template

```markdown
## Code Review: [Feature / Branch Name]

### Summary
[1-2 sentences: what the change does and overall quality]

### Stats
- Files changed: X
- Critical issues: X
- Warnings: X
- Suggestions: X

### Critical Issues (Must Fix)

1. 🔴 [Security] — Missing authorization check
   - **File**: `HADV.OMS.CentralAPI/Endpoints/Cart/GetCart.cs` (line 12)
   - **Issue**: Endpoint lacks [Authorize]. Any unauthenticated caller can access cart data.
   - **Fix**: Add `[Authorize]` above `[HttpPost(...)]`.

### Warnings (Should Fix)

1. 🟡 [Architecture] — Raw LINQ in service, bypassing Specification
   - **File**: `HADV.OMS.Core/Services/OrderService.cs` (line 67)
   - **Issue**: `_orderRepo.ListAsync()` called with an inline lambda instead of a Specification.
   - **Fix**: Create `Core/Entities/Specifications/Order/GetOrdersByLocation.cs`
     and use it here.

### Suggestions (Nice to Have)

1. 🔵 [Naming] — var used where explicit type is clearer
   - **File**: `HADV.OMS.Core/Services/MenuService.cs` (line 23)
```

## Step 4: Act on Findings

### Decision Tree

```
Reviewing YOUR OWN code?
│
├── YES → Fix everything yourself before requesting review
│   ├── 🔴 Critical → Fix now, do not open PR without it
│   ├── 🟡 Warning  → Fix now if quick; otherwise create a task
│   └── 🔵 Suggestion → Fix if time allows
│
└── NO → Reviewing someone else's code
    ├── 🔴 Critical → Block the PR, explain why, show the fix
    ├── 🟡 Warning  → Request changes, suggest the fix
    └── 🔵 Suggestion → Leave a comment, approve the PR
```

### After Fixing

```
☐ Re-run the review on the fixed code
☐ Verify the fix didn't introduce new issues
☐ Run dotnet build — no warnings
☐ Check the affected endpoint manually via Swagger (dev environment)
☐ Update the review report with resolution status
```

## Quick Reference — Red Flags by File Type

### Endpoint (CentralAPI/Endpoints/)
```
☐ Missing [Authorize] on a data endpoint
☐ No [SwaggerOperation] annotation
☐ Route not defined via ApiEndPoints constant
☐ Request class not in a separate .Request.cs file
☐ Using Ok() / BadRequest() instead of this.ToActionResult(result)
☐ Business logic inside the endpoint (belongs in a service)
☐ Direct DbContext or repository injection (use service interface)
```

### Service (Core/Services/)
```
☐ Method returns void or T instead of Result<T>
☐ Guard.Against.* missing for non-nullable parameters
☐ Repository queried with inline LINQ instead of Specification
☐ CancellationToken parameter missing from async method
☐ Interface not registered in CoreExtensions
☐ Implementation class not marked internal
☐ Direct DbContext injection (use IRepository<T>)
```

### Specification (Core/Entities/Specifications/)
```
☐ Class not marked internal
☐ Wrong aggregate root folder
☐ Business logic inside the specification
☐ .Include() used in a projection spec (redundant — use .Select())
☐ Filtering on IsDeleted manually (global filter already handles it)
```

### EF Configuration (Infrastructure/Data/Configurations/)
```
☐ base.Configure(builder) not called, or called before custom config
☐ Schema set to a raw string instead of Constants.TableSchemas.*
☐ Missing index on a column used in frequent WHERE filters
☐ New entity not added as DbSet in AppDbContext
☐ Migration not created after schema change
☐ Nullable column that should be required (or vice versa)
```

### DTO (Core/DTO/)
```
☐ Inherits from BaseEntity or another class (should be pure POCO)
☐ String property not initialized to string.Empty
☐ Collection not initialized with new() when it should never be null
☐ Sensitive data included that should be excluded from the response
```

### Entity (Core/Entities/)
```
☐ Does not extend BaseEntity (or LookupBaseEntity for lookups)
☐ Aggregate root missing IAggregateRoot marker
☐ [MaxLength] missing on string properties
☐ JSON column missing [Column(TypeName = "json")]
☐ Business logic in constructor (entities are data containers)
```
